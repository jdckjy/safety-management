## 3. 핵심 아키텍처

본 애플리케케이션은 다음과 같은 핵심 아키텍처 패턴을 따릅니다.

- **컴포넌트 기반 아키텍처 (CBA)**: UI를 독립적이고 재사용 가능한 컴포넌트로 분리하여 개발 및 유지보수를 용이하게 합니다.
- **상태 관리**: React Context API를 활용한 Provider 패턴을 사용하여 전역 상태(인증, 알림, 프로젝트 데이터 등)를 효율적으로 관리합니다.
- **라우팅**: 사용자의 인증 상태에 따라 조건부 렌더링을 통해 페이지를 전환하고, `MainLayout` 내에서는 `navigationState`를 기반으로 동적으로 콘텐츠를 표시합니다.

### 3.1. 주요 Provider

- `ProjectDataProvider`: 앱의 핵심 데이터 및 상태 관리를 총괄합니다. (상세 내용은 `7.1.` 참조)
- `NotificationProvider`: 앱 내 알림 시스템을 관리합니다.
- `SearchProvider`: 검색 기능을 위한 상태와 로직을 관리합니다.

### 3.2. Headless Floor Plan 아키텍처 (데이터 주도형 UI) - 신규 도입!

- **개념**: 과거에는 유닛의 도면 형태(SVG 경로)를 `floorPlanLayouts.ts` 라는 별도의 코드 파일에 하드코딩하여 관리했습니다. **Headless 아키텍처**는 이 구조를 완전히 뒤집어, 모든 SVG `pathData`를 `TenantUnit` 객체의 속성으로서 데이터베이스에 직접 저장하고 관리하는 방식입니다.
- **핵심 변화**: 이제 UI 컴포넌트(`FloorPlan.tsx`, `FloorPlanDrafter.tsx` 등)는 도면의 '모양'을 알지 못합니다. 그저 데이터베이스로부터 받은 `pathData` 문자열을 화면에 '그려주기만' 하는 순수한 데이터 소비/생산자가 되었습니다.
- **기대 효과**:
    - **유지보수 비용 제로**: 과거에는 도면 수정 시 개발자의 코드 수정과 재배포가 필수였으나, 이제는 관리자가 '도면 편집기'에서 직접 수정하고 저장하면 즉시 모든 서비스에 반영됩니다. 개발자의 개입이 전혀 필요 없습니다.
    - **유연한 확장성**: 신규 건물을 추가하거나 기존 층의 레이아웃을 완전히 변경할 때, 코드 수정 없이 데이터베이스에 새로운 유닛 정보와 `pathData`를 추가하는 것만으로 시스템 확장이 가능합니다.
    - **안정성 및 일관성**: 모든 도면 데이터의 출처가 데이터베이스로 단일화되어, 데이터 불일치 문제가 원천적으로 차단됩니다.

## 7. Provider 분석

### 7.1. ProjectDataProvider

앱의 핵심 데이터(KPI, 활동, 업무 등)와 관련된 모든 상태 및 비즈니스 로직을 포함하는 가장 중요한 Provider입니다. Firestore 데이터베이스와의 통신을 담당하며, 데이터 CRUD(생성, 읽기, 업데이트, 삭제)를 위한 함수들을 UI 컴포넌트에 제공합니다.

**주요 함수:**
- `addTask`: 새로운 업무를 추가합니다.
- `updateTask`: 기존 업무를 수정합니다.
- `deleteTask`: 업무를 삭제합니다.
- `addCommentToTask` **(신규)**: 특정 업무에 댓글을 추가합니다. 댓글 내용과 작성자, 타임스탬프를 포함하는 객체를 생성하여 해당 업무의 `comments` 배열에 추가하고, 변경사항을 데이터베이스에 저장합니다.
- `updateTenantUnit`: 임대 유닛 정보(도면 데이터 포함)를 업데이트합니다.
- `deleteActivityFromKpi`: KPI에서 특정 활동을 삭제합니다.

## 8. 동적 도면 관리 시스템 (구: 도면 편집기)

### 8.1. 개요
웹 애플리케이션 내에서 **데이터베이스와 직접 연동하여** 도면 정보를 시각적으로 생성 및 수정하는 관리 도구입니다. 사용자가 그린 다각형 경계는 SVG `pathData` 문자열로 변환되어, 해당 유닛의 정보와 함께 데이터베이스에 직접 저장됩니다. 이 데이터는 임대 현황판 등 시스템 전반에서 활용됩니다.

### 8.2. 핵심 컴포넌트
- **`TenantRoster.tsx`**: 메인 임대 현황 관리 화면. '도면 편집' 버튼을 통해 편집기 모드로 진입합니다.
- **`FloorPlanDrafter.tsx`**: 실제 도면 편집이 이루어지는 컴포넌트.
    - 층(1F, 2F, 3F)과 편집할 유닛을 선택합니다.
    - 마우스 클릭으로 점을 찍어 다각형을 그리고, '저장' 버튼을 누르면 **`useProjectData` 훅의 `updateTenantUnit` 함수를 호출**하여 변경사항을 데이터베이스에 직접 저장합니다.
- **`FloorPlan.tsx`**: 저장된 다각형(`pathData`)을 도면 이미지 위에 렌더링하는 순수한 뷰(View) 컴포넌트입니다.

### 8.3. 데이터 흐름 (개선된 아키텍처)
1.  **편집**:
    - `FloorPlanDrafter.tsx`에서 사용자가 특정 유닛을 선택하고 경계를 그린 후 '저장' 버튼을 클릭합니다.
    - 그려진 좌표(`points`)는 SVG `pathData` 문자열로 변환됩니다.
    - `updateTenantUnit` 함수가 호출되어, `pathData`가 포함된 `TenantUnit` 객체를 `ProjectDataProvider`로 전달합니다.
    - `ProjectDataProvider`는 내부 상태(`data`)를 업데이트하고, 이 변경은 `useEffect` 훅을 통해 **자동으로 Firestore 데이터베이스에 동기화**됩니다.
2.  **조회**:
    - `FloorPlan.tsx` 컴포넌트는 `props`로 `tenantUnits` 배열을 전달받습니다.
    - 컴포넌트는 배열을 순회하며 각 `unit` 객체의 `pathData` 속성을 읽어 SVG `path` 태그의 `d` 속성에 바인딩하여 화면에 렌더링합니다.
    - **과거에 참조했던 `floorPlanLayouts.ts` 파일은 이제 존재하지 않으며, 컴포넌트는 오직 DB 데이터에만 의존합니다.**

## 9. 작업 기반 댓글 기능 (신규)

### 9.1. 개요
단순한 업무 목록을 넘어, 각 업무(Task)의 상세 맥락과 히스토리를 추적하고 팀원 간의 소통을 활성화하기 위해 도입된 기능입니다. 사용자는 이제 각 업무별로 댓글을 작성하고, 다른 사람들이 남긴 댓글을 시간 순으로 확인할 수 있습니다.

### 9.2. 핵심 컴포넌트 및 데이터 모델
- **`WeeklyPerformanceModal.tsx`**: 기능의 핵심 UI가 구현된 곳입니다.
    - 주간 업무 목록 뷰에서 각 업무 항목 옆에 **댓글 아이콘과 현재 댓글 수를 표시**하여 새로운 소통 내용을 쉽게 인지할 수 있도록 개선되었습니다. (댓글이 있는 경우에만 표시)
    - 업무 항목을 클릭하면, 해당 업무의 상세 정보와 댓글을 볼 수 있는 '댓글 뷰(Comments View)'로 전환됩니다.
    - 댓글 뷰에서는 댓글 목록을 스크롤하여 확인하고, 하단의 입력 폼을 통해 새로운 댓글을 등록할 수 있습니다.
    - '업무 목록으로 돌아가기' 버튼을 통해 다시 목록 뷰로 돌아갈 수 있습니다.
- **`ProjectDataProvider.tsx`**: 데이터 처리를 담당합니다.
    - `addCommentToTask` 함수가 새롭게 추가되었습니다. 이 함수는 KPI ID, 활동 ID, 업무 ID 및 댓글 내용을 인자로 받아 새로운 댓글 객체를 생성하고, Firestore 데이터베이스의 해당 업무 문서(document) 내 `comments` 배열에 추가합니다.
- **`types.ts`**: 데이터 모델이 확장되었습니다.
    - `Task` 인터페이스에 `comments` 속성이 추가되었습니다. 이 속성은 `Comment[]` 타입의 배열로, 각 댓글은 `id`, `author`, `content`, `timestamp`를 포함합니다.

### 9.3. 데이터 흐름
1.  **댓글 작성**:
    - 사용자가 `WeeklyPerformanceModal.tsx`의 댓글 입력 폼에 내용을 작성하고 '전송(Send)' 버튼을 클릭합니다.
    - `handleAddComment` 이벤트 핸들러가 `useProjectData` 훅에서 제공하는 `addCommentToTask` 함수를 호출합니다.
2.  **데이터 업데이트**:
    - `ProjectDataProvider`의 `addCommentToTask` 함수는 먼저 로컬 상태(`kpiData`)를 즉시 업데이트하여 UI에 낙관적 업데이트(Optimistic Update)를 반영합니다.
    - 그 다음, `updateKpiDocInFirestore` 함수를 호출하여 변경된 데이터 전체를 Firestore 데이터베이스에 비동기적으로 저장합니다. 이를 통해 데이터의 일관성과 영속성을 보장합니다.
3.  **조회**:
    - `WeeklyPerformanceModal`이 렌더링될 때, `liveSelectedTask` 객체(실시간 데이터가 반영된 Task)의 `comments` 배열을 읽어 화면에 댓글 목록을 표시합니다. 데이터는 Provider를 통해 항상 최신 상태로 유지됩니다.
