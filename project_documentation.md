
# 단지조성팀 업무관리 시스템 기술 문서

## 1. 프로젝트 개요

본 문서는 단지조성팀의 KPI 기반 업무 관리 시스템의 기술적인 아키텍처, 핵심 기능, 데이터 흐름 등을 상세히 설명하여 신규 참여자의 이해를 돕고 유지보수의 효율성을 높이는 것을 목적으로 합니다.

## 2. 핵심 아키텍처 및 디자인 패턴

- **Component-Based Architecture**: React를 사용하여 모든 UI를 재사용 가능한 독립적인 컴포넌트로 분리하여 개발합니다.
- **State Management**: 상태 관리는 React Context API를 기반으로 한 커스텀 Provider(`ProjectDataProvider`)를 통해 중앙에서 관리됩니다. 이를 통해 상태의 일관성을 유지하고, 컴포넌트 간의 데이터 전달을 용이하게 합니다.
- **Headless Architecture**: 도면 데이터(SVG Path)를 UI 코드에서 분리하여 Firestore 데이터베이스에서 직접 관리합니다. (상세 내용은 `3. Headless 아키텍처` 참조)
- **Utility-First CSS**: Tailwind CSS를 사용하여 디자인 시스템을 구축하고, 일관되고 빠른 스타일링을 구현합니다.

## 3. Headless 아키텍처: 동적 도면 관리 시스템

### 3.1. 문제 정의
과거에는 상가(유닛)의 도면 형태를 나타내는 SVG 경로 데이터가 `floorPlanLayouts.ts`와 같은 소스코드 파일 내에 하드코딩되어 있었습니다. 이 방식은 다음과 같은 심각한 문제를 야기했습니다.

-   **높은 유지보수 비용**: 도면의 작은 수정이라도 발생하면, 항상 개발자의 코드 수정, 테스트, 그리고 재배포 과정이 필요했습니다.
-   **유연성 부족**: 새로운 층이나 건물을 추가할 때마다 시스템 전체를 다시 빌드하고 배포해야 했습니다.
-   **확장성 제한**: 코드와 데이터가 강하게 결합되어 있어, 외부 시스템과의 연동이나 기능 확장이 어려웠습니다.

### 3.2. 해결 방안: Headless 아키텍처 도입
- **개념**: 과거에는 유닛의 도면 형태(SVG 경로)를 `floorPlanLayouts.ts` 라는 별도의 코드 파일에 하드코딩하여 관리했습니다. **Headless 아키텍처**는 이 구조를 완전히 뒤집어, 모든 SVG `pathData`를 `TenantUnit` 객체의 속성으로서 데이터베이스에 직접 저장하고 관리하는 방식입니다.
- **핵심 변화**: 이제 UI 컴포넌트(`FloorPlan.tsx`, `FloorPlanDrafter.tsx` 등)는 도면의 '모양'을 알지 못합니다. 그저 데이터베이스로부터 받은 `pathData` 문자열을 화면에 '그려주기만' 하는 순수한 데이터 소비/생산자가 되었습니다.
- **기대 효과**:
    - **유지보수 비용 제로**: 과거에는 도면 수정 시 개발자의 코드 수정과 재배포가 필수였으나, 이제는 관리자가 '도면 편집기'에서 직접 수정하고 저장하면 즉시 모든 서비스에 반영됩니다. 개발자의 개입이 전혀 필요 없습니다.
    - **유연한 확장성**: 신규 건물을 추가하거나 기존 층의 레이아웃을 완전히 변경할 때, 코드 수정 없이 데이터베이스에 새로운 유닛 정보와 `pathData`를 추가하는 것만으로 시스템 확장이 가능합니다.
    - **안정성 및 일관성**: 모든 도면 데이터의 출처가 데이터베이스로 단일화되어, 데이터 불일치 문제가 원천적으로 차단됩니다.

## 7. Provider 분석

### 7.1. ProjectDataProvider

앱의 핵심 데이터(KPI, 활동, 업무 등)와 관련된 모든 상태 및 비즈니스 로직을 포함하는 가장 중요한 Provider입니다. Firestore 데이터베이스와의 통신을 담당하며, 데이터 CRUD(생성, 읽기, 업데이트, 삭제)를 위한 함수들을 UI 컴포넌트에 제공합니다.

### 7.2. AuthProvider

Firebase Authentication을 사용한 사용자 인증 관련 로직(로그인, 로그아웃, 사용자 정보 추적)을 처리합니다.

## 8. 동적 도면 데이터 아키텍처 (심화)

### 8.1. 데이터 모델

-   **`TenantUnit` (in Firestore)**
    -   `id`: `string`
    -   `name`: `string`
    -   `tenant`: `string`
    -   `pathData`: `string` (SVG `<path>` 태그의 `d` 속성에 들어갈 좌표 데이터)
    -   ...기타 메타데이터

### 8.2. 핵심 컴포넌트
-   `FloorPlanDrafter.tsx`: 관리자가 SVG `<path>`를 시각적으로 그리고 수정할 수 있는 **도면 편집기** 컴포넌트.
-   `FloorPlan.tsx`: Firestore로부터 `TenantUnit` 데이터를 받아와 `pathData`를 기반으로 실제 도면을 화면에 렌더링하는 **도면 뷰어** 컴포넌트.

### 8.3. 데이터 흐름 (개선된 아키텍처)
1.  **편집**:
    - `FloorPlanDrafter.tsx`에서 사용자가 특정 유닛을 선택하고 경계를 그린 후 '저장' 버튼을 클릭합니다.
    - 그려진 좌표(`points`)는 SVG `pathData` 문자열로 변환됩니다.
    - `updateTenantUnit` 함수가 호출되어, `pathData`가 포함된 `TenantUnit` 객체를 `ProjectDataProvider`로 전달합니다.
    - `ProjectDataProvider`는 내부 상태(`data`)를 업데이트하고, 이 변경은 `useEffect` 훅을 통해 **자동으로 Firestore 데이터베이스에 동기화**됩니다.
2.  **조회**:
    - `FloorPlan.tsx` 컴포넌트는 `props`로 `tenantUnits` 배열을 전달받습니다.
    - 컴포넌트는 배열을 순회하며 각 `unit` 객체의 `pathData` 속성을 읽어 SVG `path` 태그의 `d` 속성에 바인딩하여 화면에 렌더링합니다.
    - **과거에 참조했던 `floorPlanLayouts.ts` 파일은 이제 존재하지 않으며, 컴포넌트는 오직 DB 데이터에만 의존합니다.**

## 9. 작업 기반 댓글 기능 (신규)

### 9.1. 개요
단순한 업무 목록을 넘어, 각 업무(Task)의 상세 맥락과 히스토리를 추적하고 팀원 간의 소통을 활성화하기 위해 도입된 기능입니다. 사용자는 이제 각 업무별로 댓글을 작성하고, 다른 사람들이 남긴 댓글을 시간 순으로 확인할 수 있습니다.

### 9.2. 데이터 모델
- **`Comment` (in `Task` object)**
    - `id`: `string` (UUID)
    - `author`: `string` (작성자 이름)
    - `content`: `string` (댓글 내용)
    - `timestamp`: `string` (ISO 8601 형식의 타임스탬프)

### 9.3. 데이터 흐름
1.  **댓글 조회**: `WeeklyPerformanceModal`에서 특정 업무 선택 시, 해당 업무 객체(`Task`)에 포함된 `comments` 배열을 읽어 화면에 렌더링합니다.
2.  **댓글 작성**:
    - 사용자가 댓글을 입력하고 '전송' 버튼을 클릭합니다.
    - `addCommentToTask` 함수가 (`kpiId`, `activityId`, `taskId`, `commentText`)를 인자로 받아 호출됩니다.
    - `ProjectDataProvider`는 해당 Task를 찾아 `comments` 배열에 새로운 댓글 객체를 추가하고, 변경된 전체 프로젝트 데이터를 Firestore에 업데이트합니다.

## 10. 다중 담당자 기능 및 안정성 개선 (신규)

### 10.1. 개요
기존에는 하나의 업무(Task)에 한 명의 담당자만 지정할 수 있었지만, 여러 명이 협업하는 업무를 관리하기 어렵다는 한계가 있었습니다. 이를 해결하고, 시스템 전반의 안정성을 높이기 위해 대대적인 개선 작업을 진행했습니다.

### 10.2. 핵심 변경사항

**1. 데이터 모델 확장: 다중 담당자(Multi-assignee) 지원**
-   **`Task` 객체 구조 변경**: 기존의 단일 담당자를 의미하는 `assignee` 필드를 제거하고, 여러 명의 담당자 정보를 담을 수 있는 `assignees: TeamMember[]` 배열 필드를 추가했습니다.
-   **UI/UX 개선**: 칸반 보드(`TaskBoard.tsx`)와 주간 보고 모달(`WeeklyPerformanceModal.tsx`)의 업무 카드 UI를 개선하여, 여러 담당자의 프로필 사진이 겹쳐서 표시되도록 변경했습니다. 이를 통해 담당자 정보를 한눈에 파악할 수 있습니다.

**2. 전역 업무 수정 모달(`TaskEditModal`) 아키텍처 개선**
-   **문제점**: 서로 다른 컨텍스트(칸반 보드, 주간 보고 모달)에서 `TaskEditModal`을 호출할 때, 상위 정보(KPI, Activity)가 누락되는 치명적인 버그가 발생했습니다.
-   **해결**: `TaskEditModal`을 호출하는 모든 지점에서 **반드시 `kpiId`와 `activityId`를 명시적인 `props`로 전달하도록 계약을 변경**하고, `TaskEditModal` 내부에서는 이 `props`를 받아 상위 데이터를 정확히 찾도록 로직을 수정했습니다.
-   **기대 효과**:
    -   **안정성 확보**: 데이터 누락으로 인한 앱 충돌 및 오작동을 원천적으로 방지합니다.
    -   **컴포넌트 독립성 증대**: `TaskEditModal`이 더 이상 자신이 어떤 컨텍스트에서 호출되었는지 신경 쓸 필요 없이, 주어진 `props`만으로 독립적으로 동작할 수 있게 되었습니다.

### 10.3. 새로운 데이터 흐름 (업무 수정)
1.  사용자가 칸반 보드 또는 주간 보고 모달에서 특정 업무의 '수정' 버튼을 클릭합니다.
2.  해당 UI 컴포넌트(`TaskBoard` 또는 `WeeklyPerformanceModal`)는 자신이 알고 있는 `task` 객체, `kpiId`, `activityId`를 모두 `TaskEditModal`의 `props`로 전달하며 모달을 호출합니다.
3.  `TaskEditModal`은 전달받은 `props`를 사용하여 초기 상태를 설정하고, 사용자로부터 담당자 변경 등 수정사항을 입력받습니다.
4.  사용자가 '저장'을 클릭하면, `TaskEditModal`은 `useProjectData` 훅의 `updateTask` 함수를 호출합니다. 이때, **수정된 `task` 객체와 함께 `kpiId`, `activityId`를 정확히 전달**합니다.
5.  `ProjectDataProvider`는 이 정보들을 바탕으로 올바른 위치의 데이터를 업데이트하고, Firestore와 동기화합니다.

## 11. 프로젝트 분석 대시보드 및 UI 개선 (신규)

### 11.1. 개요
프로젝트의 전반적인 상태를 직관적으로 파악하고, 데이터 기반의 선제적 대응을 가능하게 하기 위해 관리자용 '프로젝트 분석 대시보드'를 신설했습니다. 또한, 사용자 경험을 개선하기 위해 기존 UI의 일부를 리팩토링했습니다.

### 11.2. 핵심 기능 및 변경사항

**1. 프로젝트 분석 대시보드 (`AnalysisDashboard.tsx`)**
- **목적**: 프로젝트의 핵심 지표(KPI 달성률, 병목 현상, 팀원별 업무량)를 시각적으로 요약하여 관리자가 프로젝트의 '건강 상태'를 한눈에 진단할 수 있도록 지원합니다.
- **주요 기능**:
    - **KPI 달성률 예측**: 현재 추세를 기반으로 최종 KPI 달성률을 예측합니다.
    - **병목 현상 감지**: 지연되거나 정체 상태에 있는 주요 업무를 식별하여 개수를 표시합니다.
    - **팀원별 업무량 분석**: 주요 담당자 간의 업무 분배가 균형 있게 이루어지고 있는지 분석합니다.
- **핵심 컴포넌트**:
    - `AnalysisDashboard.tsx`: 전체 대시보드의 레이아웃과 데이터 시각화를 담당합니다.
    - `useAnalytics.ts`: `ProjectDataProvider`로부터 데이터를 받아, 대시보드에 필요한 분석 지표(예측, 병목 업무 필터링 등)를 계산하는 비즈니스 로직을 포함한 커스텀 훅입니다.

**2. 병목 업무 목록 모달 (`BottleneckTasksModal.tsx`)**
- **UX 개선**: 기존에는 '주의가 필요한 업무 목록'이 대시보드 하단에 항상 노출되어, 목록이 길어질 경우 레이아웃이 깨지는 문제가 있었습니다.
- **해결**: 이 문제를 해결하기 위해, 대시보드의 '병목 현상 감지' 카드를 클릭했을 때만 상세 목록이 모달 창으로 나타나도록 구조를 변경했습니다.
- **데이터 흐름**:
    1. `AnalysisDashboard`가 '병목 현상 감지' 카드의 클릭 이벤트를 감지합니다.
    2. `useState`로 관리되는 모달 표시 상태(`isModalOpen`)를 `true`로 변경합니다.
    3. `BottleneckTasksModal` 컴포넌트는 `isOpen` 상태에 따라 화면에 표시되며, `props`로 전달받은 병목 업무 목록(`bottleneckTasks`)을 렌더링합니다.

**3. '오늘의 브리핑' UI 리팩토링 (`DailyBriefing.tsx`)**
- **문제점**: 관리자에게 표시되는 '오늘의 브리핑' 모달에 분석 대시보드 내용이 중복으로 포함되어 정보의 밀도가 너무 높고 비효율적이었습니다.
- **해결**: '오늘의 브리핑'은 이름 그대로 **그날의 핵심 업무**에만 집중할 수 있도록, 분석 대시보드 관련 섹션을 과감히 제거했습니다.
- **기대 효과**: 사용자가 매일 아침 확인하는 브리핑 모달이 더욱 간결하고 명확해져, 정보의 피로도를 줄이고 중요한 내용에만 집중할 수 있게 되었습니다.
