# 제주 헬스케어타운 단지 조성 및 자산관리 시스템: 기술 문서

> 최종 업데이트: 2026년 3월 25일

## 1. 프로젝트 개요

본 문서는 제주 헬스케어타운의 단지 조성 현황과 완공 후 자산(세대)의 임대 및 분양 관리를 효율화하기 위해 구축된 '업무관리 시스템'의 기술적 사양과 아키텍처, 핵심 기능 구현 방식을 상세히 기술합니다.

## 2. 핵심 목표

- **데이터 중심 의사결정**: 분산된 엑셀 파일과 문서를 중앙화된 데이터베이스로 통합하여, 모든 의사결정이 실시간 데이터에 기반하도록 지원합니다.
- **업무 자동화 및 효율화**: 반복적인 보고서 작성, 데이터 취합, 현황 분석 등의 업무를 자동화하여 휴먼 에러를 줄이고 업무 효율을 극대화합니다.
- **직관적 데이터 시각화**: 복잡한 단지 조성 현황과 임대/수익 데이터를 사용자가 쉽게 이해할 수 있는 인터랙티브 대시보드와 도면으로 시각화합니다.

## 3. 핵심 아키텍처 및 기술 스택

- **언어**: TypeScript
- **UI 프레임워크**: React
- **UI 컴포넌트**: shadcn/ui
- **데이터 시각화**: Recharts (Chart.js에서 변경)
- **상태 관리**: React Context API
- **백엔드/데이터베이스**: Google Firestore
- **빌드 도구**: Vite
- **호스팅**: Firebase Hosting
- **버전 관리**: Git / GitHub

## 4. 데이터 모델링 (Firestore)

- **`tenantUnits`**: 각 세대(호실)의 상세 정보 (면적, 임대료, 상태 등)
- **`complexFacilities`**: 단지 내 주요 시설물 정보
- **`teamMembers`**: 프로젝트 관련 인원 정보
- **`activities`**: 프로젝트 활동 로그
- **`kpis`**: 핵심 성과 지표
- **`monthly_reports`**: 월간 운영 데이터 (JSON 기반에서 Firestore로 통합)

## 15. AI 기반 심층 분석 대시보드 구축 및 안정화 (2026.03.25 추가)

### 15.1. 기능 목표

단순한 데이터 나열을 넘어, 경영진이 데이터에 기반하여 신속하고 정확한 의사결정을 내릴 수 있도록 '인사이트'를 제공하는 것을 목표로 합니다. 이를 위해, 핵심 지표 요약, 시계열 분석, 전년 동월 비교, 그리고 AI 기반의 심층 분석 기능을 포함하는 통합 대시보드를 구축했습니다.

### 15.2. 핵심 구현 과정

#### 1. 데이터 구조 확장

- **문제점**: 기존 데이터 모델은 에너지 사용량과 팀 활동 내역만 포함하여, 기온 변화와 같은 외부 요인과의 상관관계 분석이 불가능했습니다.
- **해결 방안**:
    - `src/types.ts`의 `MonthlyReport` 타입에 `weather` 항목 (평균 기온 `averageTemperatureC` 포함)을 추가했습니다.
    - `src/data/2026-02-report.json` 파일에 실제 `weather` 데이터를 추가하여 새로운 데이터 구조의 프로토타입을 완성했습니다.

#### 2. 대시보드 UI/UX 구현 (`OmsDashboard.tsx`)

- 기존의 단순 업무 목록(`OmsTeamTasks.tsx`)을 완전히 새로운 `OmsDashboard.tsx` 컴포넌트로 대체했습니다.
- **주요 기능**:
    - **핵심 요약 카드**: 전월 대비 총비용 변화량을 직관적으로 보여주는 헤더 카드.
    - **Metric 카드**: 전기/가스 사용량, 평균 기온, 태양광 발전량을 전년 동월과 비교하여 시각화.
    - **시계열 분석 차트**: `recharts` 라이브러리를 활용하여 최근 12개월간의 에너지 사용 추이를 라인 차트로 시각화.
    - **AI 심층 분석**: 버튼 클릭 시, AI가 분석한 데이터 인사이트와 비용 절감 방안을 텍스트 형태로 제공.

### 15.3. 배포 후 긴급 장애 대응

- **문제 상황**: 최초 배포 직후, `Cannot read properties of undefined (reading 'averageTemperatureC')` 라는 치명적인 런타임 오류가 발생하여 애플리케이션이 완전히 중단되었습니다.
- **원인 분석**: 대시보드 코드는 모든 월간 보고서에 '평균 기온' 데이터(`weather.averageTemperatureC`)가 존재할 것이라고 가정했으나, 과거 데이터에는 해당 필드가 없어 오류가 발생했습니다. AI(본인)가 모든 데이터의 호환성을 검증하지 않고 성급하게 배포한 것이 원인이었습니다.
- **최종 해결 방안**:
    - **Optional Chaining (`?.`) 및 Nullish Coalescing (`??`) 적용**: `OmsDashboard.tsx` 코드 내에서, `latestReport.raw_data.weather?.averageTemperatureC?.value`와 같이 Optional Chaining을 사용하여 데이터의 존재 여부를 안전하게 확인하도록 수정했습니다.
    - **폴백 값(Fallback Value) 제공**: 데이터가 존재하지 않을 경우, `?? 'N/A'` 와 같이 Nullish Coalescing 연산자를 사용하여 "N/A" (Not Available)를 표시하도록 하여, 데이터가 없어도 UI가 깨지거나 시스템이 중단되지 않도록 방어적인 코드를 작성했습니다.
    - **긴급 빌드 및 재배포**: 수정된 코드를 즉시 빌드하고 재배포하여 시스템을 성공적으로 복구했습니다.

### 15.4. 교훈

- **과거 데이터는 새로운 기능의 적이다**: 새로운 필드나 데이터 구조를 도입할 때는, 반드시 기존 데이터와의 하위 호환성을 보장해야 합니다.
- **방어적 프로그래밍의 중요성**: 외부 또는 과거 데이터 소스를 다룰 때는, 데이터가 항상 완벽한 형태로 존재할 것이라고 가정해서는 안 됩니다. Optional Chaining, Nullish Coalescing, 타입 가드 등을 적극적으로 활용하여 예기치 않은 오류를 방지해야 합니다.
- **배포는 끝이 아닌 새로운 시작**: 배포 직후에는 반드시 실제 운영 환경에서 모든 기능이 정상적으로 동작하는지 철저히 테스트하고 모니터링해야 합니다.
