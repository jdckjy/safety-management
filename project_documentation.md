
# 프로젝트 문서

## 1. 개요

이 문서는 React 기반 웹 애플리케이션의 전체 구조, 주요 컴포넌트, 데이터 흐름 및 핵심 아키텍처에 대해 상세히 설명합니다.

## 2. AI 협업 가이드라인

이 섹션은 AI 어시스턴트와의 효율적이고 안전한 협업을 위해 정의된 규칙입니다.

### 2.1. 기본 원칙 (Core Principles)

- **추측 금지 (No Hallucination)**: AI는 확실하지 않은 파일 경로, 변수명, 비즈니스 로직에 대해 절대 추측하지 않고, 반드시 사용자에게 질문하여 명확한 지침을 받습니다.
- **시니어 시각 (Senior Perspective)**: AI는 20년 경력의 시스템 설계자 관점에서, 코드의 확장성과 유지보수 용이성을 최우선 가치로 삼고 개발에 임합니다.
- **영향도 분석 (Impact Analysis)**: 코드 수정 전, 제안하는 변경이 기존의 다른 기능에 미칠 수 있는 잠재적 영향을 반드시 분석하고 보고합니다.

### 2.2. 코드 수정 전 필수 체크리스트 (Pre-flight Checklist)

AI는 코드를 작성하기 전, 다음 3가지 사항을 분석하여 사용자에게 보고하고 승인을 받아야 합니다.

- **기존 기능 유실 검토**: 수정하려는 코드가 기존 로직과 어떤 의존성을 가지고 있는지, 변경 시 예상치 못한 사이드 이펙트는 없는지 확인합니다.
- **구조적 정확성**: 모든 파일 경로는 실제 파일 시스템을 기준으로 하며, import 경로는 추측이 아닌 확인된 경로만을 사용합니다.
- **유지보수성**: 하드코딩을 최대한 지양하고, 공통 Hook이나 Constants를 활용하여 미래의 요구사항 변경에 유연하게 대처할 수 있는 코드를 작성합니다.

### 2.3. 작업 프로세스 (Step-by-Step Workflow)

1.  **분석**: 사용자의 수정 요청을 받으면, 관련된 파일들을 모두 읽고 변경이 미칠 '영향 범위'를 요약하여 보고합니다.
2.  **설계**: "확장성 고려 완료, 경로 확인 완료"와 같은 선언과 함께, 구체적인 수정 방향과 계획을 제안합니다.
3.  **구현**: 사용자의 최종 승인 후 코드를 작성하며, 제출 전 기존 기능과의 충돌 여부를 마지막으로 점검합니다.

## 3. 핵심 아키텍처

본 애플리케이션은 다음과 같은 핵심 아키텍처 패턴을 따릅니다.

- **컴포넌트 기반 아키텍처 (CBA)**: UI를 독립적이고 재사용 가능한 컴포넌트로 분리하여 개발 및 유지보수를 용이하게 합니다.
- **상태 관리**: React Context API를 활용한 Provider 패턴을 사용하여 전역 상태(인증, 알림, 프로젝트 데이터 등)를 효율적으로 관리합니다.
- **라우팅**: 사용자의 인증 상태에 따라 조건부 렌더링을 통해 페이지를 전환하고, `MainLayout` 내에서는 `navigationState`를 기반으로 동적으로 콘텐츠를 표시합니다.

### 3.1. 주요 Provider

- `ProjectDataProvider`: 앱의 핵심 데이터 및 상태 관리를 총괄합니다. (상세 내용은 `7.1.` 참조)
- `NotificationProvider`: 앱 내 알림 시스템을 관리합니다.
- `SearchProvider`: 검색 기능을 위한 상태와 로직을 관리합니다.

## 7. Provider 분석

### 7.1. ProjectDataProvider (ProjectDataProvider.tsx)

- **역할**: 애플리케이션의 모든 핵심 데이터와 상태를 관리하고, 하위 컴포넌트에 데이터와 상태 조작 함수를 제공하는 최상위 Provider입니다.

- **주요 기능 및 상태**:
    - **데이터 관리**: `safetyKPIs`, `leaseKPIs`, `assetKPIs`, `infraKPIs`, `hotspots`, `facilities`, `complexFacilities`, `teamMembers` 등 애플리케이션의 거의 모든 동적 데이터를 포함하는 `data` 객체를 상태로 관리합니다.
    - **Firebase Firestore 연동**:
        - `useAuth` 훅을 통해 현재 사용자(`currentUser`) 정보를 가져옵니다.
        - 사용자가 로그인하면 Firestore DB의 `users/{currentUser.uid}` 경로에서 해당 사용자의 데이터를 비동기적으로 가져와 상태를 업데이트합니다.
        - 사용자가 없는 경우(신규 사용자) 초기 데이터를 Firestore에 생성합니다.
        - `useEffect`와 디바운싱(`setTimeout`)을 결합하여, 상태(`data`)가 변경될 때마다 자동으로 변경사항을 Firestore에 저장하여 데이터 영속성을 보장합니다.
    - **네비게이션 상태 관리**: `navigationState` 객체를 통해 현재 활성화된 메뉴(`menuKey`)나 선택된 월(`selectedMonth`) 등 UI 네비게이션과 관련된 상태를 관리하며, `navigateTo` 함수를 통해 상태 변경을 지원합니다.
    - **파생 데이터 제공**: `useMemo` 훅을 사용하여 각기 다른 KPI 배열(`safetyKPIs`, `leaseKPIs` 등)을 UI에서 사용하기 편리한 단일 배열(`kpiData`)으로 가공하여 제공합니다.

- **제공되는 주요 함수**:
    - **CRUD (Create, Read, Update, Delete) 함수**: KPI, Activity, Task, 단지 시설(`ComplexFacility`), 팀원(`TeamMember`) 등 다양한 데이터에 대한 생성, 수정, 삭제 로직을 캡슐화한 함수들(`addActivityToKpi`, `updateTask`, `deleteComplexFacility`, `addTeamMember` 등)을 제공합니다. 이를 통해 하위 컴포넌트는 데이터 구조에 대한 깊은 이해 없이 상태를 변경할 수 있습니다.
    - **상태 Setter 함수**: `setSafetyKPIs`, `setHotspots` 등 특정 데이터 조각을 직접 업데이트할 수 있는 Setter 함수들을 제공합니다.

- **`useProjectData` 훅**:
    - 컴포넌트에서 `useContext(ProjectDataContext)`를 직접 사용하는 대신, `useProjectData` 훅을 통해 컨텍스트 값에 간편하게 접근할 수 있도록 합니다. 컨텍스트가 없는 경우 에러를 발생시켜 잠재적인 버그를 방지합니다.

## 8. 도면 편집기 (Floor Plan Editor) 기능

### 8.1. 개요
사용자가 웹 애플리케이션 내에서 직접 각 층의 도면을 보고, 특정 유닛(예: 사무실, 상점)이 차지하는 공간의 경계를 다각형으로 그릴 수 있는 시각적 편집 도구입니다. 이 기능을 통해 얻은 SVG 경로 데이터는 임대 현황판, 유닛 상세 정보 등 다양한 곳에서 시각적 요소로 활용될 수 있습니다.

### 8.2. 핵심 컴포넌트
- **`TenantRoster.tsx`**: 메인 임대 현황 관리 화면입니다. '도면 편집' 버튼을 통해 편집기 모드로 진입할 수 있습니다.
- **`FloorPlanDrafter.tsx`**: 실제 도면 편집이 이루어지는 컴포넌트입니다.
    - 사용자는 이 컴포넌트 안에서 층(1F, 2F, 3F)을 선택할 수 있습니다.
    - 마우스 클릭으로 점을 찍어 다각형을 그리고, '저장' 버튼을 누르면 해당 유닛의 `pathData` 속성에 SVG 경로가 저장됩니다.
- **`FloorPlan.tsx`**: 저장된 다각형(`pathData`)을 도면 이미지 위에 렌더링하여 보여주는 컴포넌트입니다.

### 8.3. 좌표계 문제 해결 (동적 SVG `viewBox`)
- **문제점**: 초기 구현 시, 도면을 그리는 편집기와 그린 결과를 보여주는 뷰어 간의 해상도(좌표계)가 달라 위치가 어긋나는 문제가 있었습니다.
- **해결책**: SVG의 `viewBox` 속성을 이미지의 원본 크기(`naturalWidth`, `naturalHeight`)에 동적으로 맞추는 로직을 `FloorPlanDrafter`와 `FloorPlan` 양쪽에 모두 적용했습니다.
    - `handleImageLoad` 이벤트 핸들러를 통해 이미지가 로드될 때 원본 이미지의 가로/세로 크기를 가져옵니다.
    - 이 크기를 `viewBox` 값(예: `viewBox="0 0 1200 800"`)으로 설정합니다.
    - 이를 통해 SVG 좌표는 항상 화면 크기가 아닌 **원본 이미지 크기를 기준**으로 계산되므로, 어떤 해상도에서도 위치가 정확히 일치하게 됩니다.

### 8.4. 데이터 흐름
1. `TenantRoster.tsx`에서 '도면 편집' 버튼 클릭 -> `FloorPlanDrafter.tsx` 렌더링
2. `FloorPlanDrafter.tsx`에서 층과 유닛 선택 후, 도면 위에 마우스 클릭
3. 클릭된 좌표들은 SVG `viewBox`에 맞춰 변환된 후, `points` 배열에 저장됨
4. '저장' 시, `points` 배열은 `M ... L ... Z` 형태의 SVG 경로 문자열(`pathData`)로 변환됨
5. `useProjectData` 컨텍스트를 통해 해당 유닛의 `pathData`가 업데이트됨
6. 편집기 모드 종료 후 `FloorPlan.tsx`는 업데이트된 `pathData`를 읽어 도면 위에 정확한 위치에 다각형을 렌더링함

### 8.5. 고급 기능 (Advanced Features)

#### 8.5.1. 직선 자동 정렬 (Snap-to-Axis)
- **목표**: 사용자가 파워포인트와 같이 반듯한 수직/수평선을 쉽게 그릴 수 있도록 지원합니다.
- **구현**: 새로운 점을 추가할 때, 바로 이전 점과의 x, y 좌표 차이를 계산합니다.
  - x좌표 차이(`deltaX`)와 y좌표 차이(`deltaY`)를 비교하여, 둘 중 하나가 미리 설정된 임계값(`snapThreshold`)보다 작으면 해당 축으로 좌표를 강제 정렬합니다.
  - 예를 들어, `deltaY`가 임계값보다 작으면 새 점의 y좌표를 이전 점의 y좌표와 동일하게 만들어 완벽한 수평선을 만듭니다.
- **효과**: 손으로 그려도 삐뚤어짐 없는 깔끔한 직선 다각형을 만들 수 있어 도면의 정확성과 작업 효율이 크게 향상됩니다.

#### 8.5.2. 기존 도면 수정
- **목표**: 한 번 저장된 SVG 경로를 다시 불러와 수정할 수 있도록 하여, 처음부터 다시 그리는 불편함을 없앱니다.
- **구현**:
  1.  `FloorPlanDrafter`에서 수정할 유닛을 선택하면, `useEffect` 훅이 해당 유닛의 `pathData` 존재 여부를 확인합니다.
  2.  `pathData`가 존재하면, 저장된 SVG 경로 문자열(예: "M 10 10 L 100 10 L 100 100 Z")을 파싱하여 좌표 배열(예: `[10, 10, 100, 10, 100, 100]`)로 변환합니다.
  3.  변환된 좌표 배열을 편집기의 `points` 상태에 설정하여, 기존에 그린 다각형을 편집 화면에 즉시 로드합니다.
- **효과**: 사용자는 기존 영역에 점을 추가하거나, '초기화' 버튼으로 완전히 새로 그리는 등 유연하게 도면을 수정할 수 있습니다.

## 8.6. 호실 정보 수정 모달 UI/UX 개선

- **초기 문제점**: `UnitEditModal.tsx` 컴포넌트의 디자인이 세련되지 않았고, 모달 하단에 콘텐츠와 무관한 불필요한 여백이 크게 발생하는 레이아웃 문제가 있었습니다.
- **원인 분석**:
    - 초기에는 `UnitEditModal.tsx` 내부의 문제로 판단하여 해당 파일만 수정했으나 문제가 해결되지 않았습니다.
    - 근본적인 원인은 앱 전역에서 사용되는 `src/components/ui/Modal.tsx` 공용 컴포넌트의 스타일에 있었습니다.
    - `Modal.tsx`는 `h-[95vh]`라는 고정적인 높이 값을 가지고 있어, 내부 콘텐츠의 길이에 상관없이 항상 화면의 95% 높이를 차지했습니다. 또한, `items-center` 속성으로 인해 모달이 화면 중앙에 위치하여 상단 정렬이 되지 않았습니다.
- **해결책**:
    - `src/components/ui/Modal.tsx` 파일을 직접 수정하여 구조적인 문제를 해결했습니다.
        1.  **가변 높이 적용**: `h-[95vh]` 클래스를 제거하여, 모달의 높이가 내부 콘텐츠에 따라 자동으로 결정되도록 변경했습니다.
        2.  **위치 및 디자인 수정**: `items-center`를 `items-start`로 변경하고 상단 여백(`pt-20`)을 추가하여 모달이 화면 상단에 위치하도록 조정했습니다. 더불어 배경색, 모서리 둥글기 등 디자인을 개선하여 일관성을 높였습니다.
- **추가 개선 사항**:
    - `UnitEditModal.tsx` 내부적으로도 `Modal` 컴포넌트의 사용법을 올바르게 수정하고, 헤더/푸터 구조를 명확히 하여 가독성과 유지보수성을 향상시켰습니다.
- **결과**:
    - 콘텐츠 길이에 맞는 반응형 높이를 가진 모달이 구현되어 불필요한 여백 문제가 완전히 해결되었습니다.
    - 일관되고 세련된 디자인의 모달 UI가 완성되어 전반적인 사용자 경험이 향상되었습니다.

## 9. UI/UX 개선 사항

### 9.1. 탭 스타일 통일 및 가변 너비 조정

- **초기 문제점**: 각 주요 기능 페이지(`임대 모집`, `자산 관리`, `인프라 개발`, `안전 관리`) 내부에 위치한 세부 탭 메뉴들의 디자인(크기, 패딩 등)이 일관되지 않았습니다. 특히, 일부 탭은 내용 길이에 상관없이 고정된 너비를 가져 텍스트가 잘리거나 불필요한 여백이 생기는 문제가 있었습니다.

- **해결책**:
    1.  **스타일 통일**: `SafetyManagement.tsx`의 탭 디자인을 기준으로 삼아, `LeaseRecruitment.tsx`, `AssetManagement.tsx`, `InfraDevelopment.tsx` 컴포넌트의 탭 스타일(활성/비활성 색상, 폰트, 그림자 등)을 모두 통일했습니다. 각 페이지의 고유 색상(Blue, Emerald, Purple)은 유지하여 시각적 구분을 명확히 했습니다.

- **결과**:
    - 애플리케이션 전체에 걸쳐 모든 탭 컴포넌트가 일관된 디자인과 동작을 가지게 되었습니다.

### 9.3. 임대율 산정 로직 최종 수정 (Lease Rate Calculation Final Fix)

- **배경**: 임대율 현황판(`TenantRoster.tsx`)의 핵심 지표인 임대율(Lease Rate) 계산 로직에서 여러 차례 오류가 발생했습니다. 초기 로직은 '비임대' 상태의 유닛을 전체 면적(모수)에서 제외하는 방식이었으나, 여러 엣지 케이스와 타입 문제로 인해 부정확한 결과를 반환하는 문제가 지속되었습니다.

- **최종 해결책**:
    1.  **명확한 산식 확립**: 사용자(팀장)와의 논의를 통해, 모든 예외 케이스를 배제하는 가장 명확하고 직접적인 산식을 최종적으로 확립했습니다.
        > **`임대율 = 입주 면적 / (입주 면적 + 공실 면적 + 협의중 면적) * 100`**

    2.  **'제외'가 아닌 '포함' 방식의 코드 구현**: 기존의 '비임대' 상태를 필터링하여 제외하는 방식은 잠재적 오류 가능성이 있다고 판단하여, 최종적으로 코드를 **명시적으로 포함하는 방식**으로 변경했습니다.
        - `totalRentableArea`(전체 임대 가능 면적, 즉 모수)를 계산할 때, `['입주', '공실', '협의중']` 상태를 가진 유닛들의 면적만을 합산하도록 로직을 수정했습니다.
        - 이를 통해 '비임대' 상태나 미래에 추가될 수 있는 다른 어떤 상태도 계산에 영향을 주지 않는, 견고하고 직관적인 코드가 완성되었습니다.

- **기술적 수정 사항**:
    - `TenantRoster.tsx` 내의 `calculateRate` 함수에서 `rentableStatuses` 라는 배열을 `[UNIT_STATUS.OCCUPIED, UNIT_STATUS.VACANT, UNIT_STATUS.IN_DISCUSSION]` 로 정의하고, `totalRentableArea` 계산 시 이 배열에 포함된 상태의 유닛만 필터링하도록 수정했습니다.
    - 또한, 이 과정에서 발생한 미묘한 TypeScript 타입 오류(`Argument of type 'string' is not assignable to parameter of type '"공실" | "입주" | "협의중"'`)를 해결하기 위해 `rentableStatuses` 배열의 타입을 `string[]`으로 명시하여 타입 안정성을 확보했습니다.

- **결과**:
    - 임대율 계산 로직의 모호함이 완전히 제거되었고, 비즈니스 요구사항과 100% 일치하는 코드가 최종적으로 반영되었습니다.
    - 이 수정 과정은 향후 유사한 기능 개발 시, 복잡한 제외 로직보다는 명시적인 포함 로직이 더 안전하고 유지보수하기 좋다는 중요한 선례를 남겼습니다.
