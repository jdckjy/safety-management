# 프로젝트 데이터 흐름 분석

## 1. 요약

이 문서는 애플리케이션의 임차인 관리 섹션 내의 데이터 관리 및 로딩 메커니즘을 분석합니다. 목업 데이터와 사용자가 생성한 데이터가 병합되어 UI, 특히 계약 현황 보기에서 불일치가 발생하는 중대한 문제가 확인되었습니다. 이 분석에서는 근본 원인을 상세히 설명하고 데이터 무결성을 보장하기 위한 해결책을 권장합니다.

## 2. 데이터 로딩 프로세스

데이터 관리 시스템의 핵심은 `ProjectDataProvider.tsx` 컴포넌트에 있으며, `useProjectData` 훅을 활용하여 애플리케이션 전체에 데이터를 배포합니다. 프로세스는 다음과 같이 나눌 수 있습니다.

1.  **초기 상태**: 애플리케이션 시작 시 시스템은 `initialData`라는 사전 정의된 데이터 구조로 초기화되며, 여기에는 임차인, 계약 및 기타 엔티티(예: `initialTenants`, `initialContracts`)에 대한 목업 데이터가 포함됩니다.

2.  **사용자 인증**: 사용자가 로그인하면 애플리케이션은 Firestore 문서(`/users/{currentUser.uid}`)에서 기존 데이터를 확인합니다.

3.  **Firestore에서 데이터 하이드레이션**:
    - 사용자 문서가 있는 경우 애플리케이션은 Firestore에서 데이터를 로드합니다.
    - 결정적으로, Firestore 문서에 `tenantInfo` 또는 `contracts`와 같은 데이터 슬라이스가 누락된 경우 시스템은 해당 슬라이스에 대한 초기 목업 데이터로 대체합니다.

4.  **최초 로그인/신규 사용자**: 사용자 문서가 없으면 애플리케이션은 모든 목업 데이터가 포함된 `initialData`로 시드하여 새 문서를 생성합니다.

5.  **데이터 지속성**: 새 임차인 또는 계약 추가와 같이 사용자가 변경한 모든 내용은 Firestore 문서에 다시 유지됩니다. 애플리케이션은 디바운스 메커니즘과 함께 `useEffect` 훅을 사용하여 기존 문서와 병합하여 데이터를 저장합니다.

## 3. 데이터 중복의 근본 원인

중복 문제는 초기 목업 데이터와 Firestore 지속성 계층 간의 상호 작용에서 비롯됩니다.

-   **목업 데이터로 시딩**: 사용자가 처음 로그인하면 Firestore 문서는 로컬 파일(예: `initial-tenants.ts`, `initial-contracts.ts`)의 전체 목업 데이터 세트로 채워집니다.

-   **누적 데이터 작업**: 사용자가 새 정보(예: 새 계약)를 추가하면 이 새 데이터가 컴포넌트 상태의 기존 배열에 추가됩니다. 이 결합된 배열(목업 및 새 데이터 모두 포함)은 Firestore에 다시 저장됩니다.

-   **주기적인 중복**: 다음에 사용자가 애플리케이션을 로드하면 이 결합된 데이터를 가져옵니다. 그런 다음 사용자가 다른 계약을 추가하면 주기가 반복되어 중복되고 오래된 목업 항목으로 데이터 세트가 더욱 확장됩니다.

이 주기는 계약 상태 목록에서 관찰된 것처럼 UI가 정확한 사용자 입력 데이터와 부정확한 목업 데이터가 혼합되어 표시되는 부풀려지고 일관성 없는 데이터 상태를 초래합니다.

## 4. 권장 해결책

이 문제를 해결하고 사용자에게 깨끗한 데이터 환경을 보장하려면 프로덕션 사용자 문서를 목업 데이터로 자동 시딩하는 관행을 중단해야 합니다. 다음 단계를 권장합니다.

1.  **목업 데이터 시딩 중단**: `ProjectDataProvider.tsx` 컴포넌트를 수정하여 생성 시 `initialData`(목업 임차인 및 계약 포함)가 사용자의 Firestore 문서에 저장되지 않도록 합니다.

2.  **데이터 마이그레이션/정리 전략 구현**:
    - **기존 사용자**의 경우 Firestore 문서에서 목업 데이터를 제거하기 위해 일회성 스크립트를 개발하고 실행해야 합니다. 이 스크립트는 사용자가 합법적으로 입력한 데이터만 보존하면서 오래된 항목을 식별하고 제거합니다.
    - **신규 사용자**의 경우 시스템은 임차인, 계약 및 관련 엔티티에 대해 진정으로 비어 있는 상태로 초기화하여 깨끗한 상태에서 데이터를 구축할 수 있도록 해야 합니다.

3.  **개발 환경 개선**: 개발 및 테스트 목적으로 목업 데이터를 계속 사용하되 프로덕션 데이터 흐름과 엄격하게 격리되도록 합니다. 이는 환경 변수 또는 기타 구성 플래그를 사용하여 비프로덕션 환경에서만 조건부로 목업 데이터를 로드하여 달성할 수 있습니다.

이러한 변경 사항을 구현함으로써 애플리케이션은 사용자가 자신의 정확한 데이터와만 상호 작용하도록 보장하여 보다 안정적이고 예측 가능한 사용자 경험을 제공할 것입니다.
