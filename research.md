
# 임대 관리 데이터 구조 분석 및 개선 방안 보고서

## 1. 분석 개요

본 문서는 현재 프로젝트의 임대 관련 데이터 관리 방식의 문제점을 진단하고, 이를 해결하기 위한 데이터베이스 구조 개선 방안을 제시하는 것을 목표로 한다. 분석 결과, 현재 시스템은 서로 다른 두 개의 데이터 모델이 공존하는 심각한 구조적 문제를 안고 있으며, 이는 데이터 불일치, 비효율적인 관리, 확장성 부재의 직접적인 원인이 되고 있다.

## 2. 현황 및 문제점 분석

현재 임대 관련 정보는 다음과 같은 두 가지 방식으로 관리되고 있다.

### A. 관계형 데이터 모델 (이상적인 설계)

- **정의**: `src/types.ts`에 `TenantInfo`, `Unit`, `Contract` 등의 인터페이스로 명확히 정의되어 있다.
- **구조**: 각 데이터(임차인, 공간, 계약)는 고유 ID를 통해 연결되는 정규화된 관계형 구조를 의도했다.
- **특징**: 데이터 중복을 최소화하고 일관성을 유지하기에 적합한 이상적인 모델이다.
- **주 사용처**: '임대 및 세대 관리' 상세 정보 뷰 (`src/features/tenant-info/`)에서 사용될 것으로 추정된다.

### B. 비정규화된 레거시 모델 (현실의 문제)

- **정의**: `src/types.ts`에 `TenantUnit` 이라는 레거시 타입으로 정의되어 있다.
- **실제 데이터**: `src/data/initial-tenant-units.ts` 파일에 모든 데이터가 하드코딩된 거대한 배열 형태로 존재한다.
- **구조**: 공간 정보(`id`, `floor`, `area`), 임차인 정보(`tenant`), 계약 상태(`status`) 등 관련 정보가 **하나의 객체에 모두 포함**된 비정규화된 구조다.
- **주 사용처**: 도면 기반의 임대 현황판인 'Tenant Roaster' 기능 (`src/features/tenant-roster/TenantRoster.tsx`) 전체가 이 데이터에 직접적으로 의존하고 있다.

### 핵심 문제점

1.  **치명적인 데이터 중복**: `initial-tenant-units.ts` 파일 내에 'KMI'와 같은 동일한 임차인 이름이 여러 번 반복해서 저장된다. 임차인 정보가 변경될 경우, 관련된 모든 레코드를 수동으로 찾아 수정해야 하므로 데이터 관리 비용이 기하급수적으로 증가하고 오류 발생 가능성이 매우 높다.
2.  **데이터 불일치 발생**: 두 개의 데이터 모델이 공존함에 따라, 한쪽 모델의 데이터만 수정될 경우 시스템 전체의 데이터 일관성이 깨지게 된다. 예를 들어, '임대 및 세대 관리'에서 임차인 이름을 변경해도 'Tenant Roaster'에는 반영되지 않는 문제가 발생한다.
3.  **확장성 및 유지보수성 저하**: 새로운 계약이 발생하거나 공간 정보가 변경될 때마다 거대한 하드코딩된 파일을 직접 수정해야 한다. 이는 매우 위험하고 비효율적인 방식으로, 시스템의 유연한 확장과 안정적인 유지보수를 불가능하게 만든다.

## 3. 개선 목표 및 실행 계획

이 모든 문제의 근본적인 해결을 위해, **두 개의 데이터 모델을 하나로 통일**하는 것을 목표로 한다. 구체적으로, 비정규화된 `TenantUnit` 레거시 모델을 완전히 제거하고, 모든 관련 기능이 `TenantInfo`, `Unit`, `Contract`의 관계형 모델을 사용하도록 시스템을 전면 리팩토링해야 한다.

### 단계별 실행 계획

#### 1단계: 데이터 정규화 (새로운 데이터 소스 생성)

- `src/data/initial-tenant-units.ts` 파일의 데이터를 기반으로, 관계형 모델에 맞는 세 개의 새로운 초기 데이터 파일을 생성한다.
    1.  **`src/data/initial-units.ts`**: 순수한 공간 정보만 포함 (`id`, `name`, `unitNumber`, `floor`, `area_sqm`, `pathData` 등)
    2.  **`src/data/initial-tenants.ts`**: 중복을 제거한 순수한 임차인 정보만 포함 (`id`, `companyName`, `businessRegistrationNumber` 등)
    3.  **`src/data/initial-contracts.ts`**: 임차인과 공간을 연결하는 계약 정보만 포함 (`id`, `tenantId`, `unitId`, `startDate`, `endDate` 등)

#### 2단계: 데이터 프로바이더 리팩토링

- `src/providers/ProjectDataProvider.tsx`를 수정한다.
- 기존의 `tenantUnits`를 로드하는 로직을 제거한다.
- 1단계에서 생성한 3개의 새로운 데이터 파일(`units`, `tenants`, `contracts`)을 로드하고, 앱 전역에서 사용할 수 있도록 제공하는 로직을 추가한다.
- 필요에 따라, 로드된 데이터를 앱에서 쉽게 사용하기 위해 메모리 상에서 데이터를 조합(join)하거나 조회(lookup)하는 헬퍼 함수를 구현한다. (예: `getTenantByUnitId(unitId)})

#### 3단계: 'Tenant Roster' 기능 전면 리팩토링

- `src/features/tenant-roster/` 디렉토리의 모든 컴포넌트(`TenantRoster.tsx`, `FloorPlan.tsx`, `UnitDetailPanel.tsx` 등)를 수정한다.
- `TenantUnit` 타입에 대한 모든 의존성을 제거한다.
- `ProjectDataProvider`로부터 새로운 데이터(`units`, `tenants`, `contracts`)를 가져와 사용하도록 로직을 변경한다.
- 예를 들어, 도면의 유닛을 렌더링할 때는 `units` 데이터를 사용하고, 유닛 선택 시 상세 정보 패널에는 `unit.tenantId`를 이용해 `tenants`와 `contracts` 데이터에서 관련 정보를 찾아 표시하도록 구현한다.

#### 4단계: 레거시 코드 제거 및 최종 검증

- `initial-tenant-units.ts` 파일을 프로젝트에서 완전히 삭제한다.
- `types.ts`에서 `TenantUnit` 타입을 삭제한다.
- 애플리케이션을 실행하여 '임대 및 세대 관리'와 'Tenant Roster' 두 기능이 동일한 데이터 소스를 참조하여 일관되게 동작하는지 철저히 검증한다.

## 4. 기대 효과

- **데이터 일관성 확보**: 단일 데이터 소스를 사용함으로써 시스템 전체의 데이터가 항상 일관성을 유지한다.
- **유지보수 효율성 증대**: 임차인, 공간, 계약 정보 변경 시 각각의 해당 데이터만 수정하면 되므로 관리가 매우 용이해진다.
- **확장성 및 유연성 향상**: 새로운 데이터 모델은 향후 다양한 형태의 임대 관련 기능 추가에 유연하게 대응할 수 있는 안정적인 기반이 된다.
- **버그 발생 가능성 감소**: 데이터 중복 및 불일치로 인해 발생하던 잠재적인 버그들을 원천적으로 차단한다.
